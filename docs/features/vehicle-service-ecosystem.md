## 1. 背景與商業價值

### 1.1 問題陳述
PCM 目前是純零件電商、客人買完零件後需自行找店家安裝、安裝資訊完全不進系統、無法形成閉環。這造成三個問題:
- 客人買完零件後 PCM 無法延伸服務、LTV 偏低
- 合作店家與 PCM 關係鬆散、沒有深度綁定的誘因
- 二手車市場缺乏可信的改裝/保養履歷、PCM 錯失平台價值延伸機會

### 1.2 本 PRD 的目標
建立一個涵蓋「買零件 → 找店家 → 預約 → 安裝 → 履歷 → 保養追蹤 → 二手車履歷」的完整閉環、讓 PCM 從「零件店」升級為「車輛服務生態平台」。

### 1.3 商業價值設計
- **PCM 只賺零件**(商品差價)、**店家只賺服務**(工資 + 額外施工)、利益不衝突
- **儲值金回饋**讓客人自然選擇合作店家、不用強迫
- **履歷只開放店家查詢**讓合作店家獲得獨家資訊價值、提高加盟誘因
- **飛輪效應**:客人越多 → 店家越多 → 商品頁越多案例 → 零件賣得越好

## 2. 目標與非目標

### 2.1 目標
- 客人下單後可自由決定何時找哪家店安裝、不被預約時機綁住
- 店家可收到清楚的詢價單、自主報價、自主接單
- 每台車都有完整的改裝 + 保養履歷、跟車不跟人
- 履歷有防偽機制(車籍驗證 + 訂單綁定 + 客人確認)
- 店家端支援手機、平板、桌機三種裝置(375px / 768px / 1920px)
- 店家老闆「不太會用電腦」的前提下仍能輕鬆使用
- 所有通訊走 Line + Email、不用簡訊

### 2.2 非目標
- 不做安裝工資線上收款(PCM 不碰這筆錢)
- 不做即時訂單取消(客訂商品一旦下訂不可取消)
- 不做客人對店家的星級評分(只做作品集 + 內部觀察)
- 初期不向店家收費(未來規模大再議)
- 不做自動仲裁(爭議一律人工處理)

## 3. 使用者流程

### 3.1 客人視角
瀏覽商品 → 下單(選收件方式、不綁預約)→ 付款 → 收到貨 → 在會員中心發詢價單 → 比較店家報價 → 選店家 → 預約時段 → 到店安裝 → 現場付工資 → 履歷記錄自動確認 → 運費退儲值金 → 保養提醒 → 下次服務

### 3.2 店家視角
申請加入(或 PCM 邀請)→ 審核通過 → 開帳號 → 每天早上 Line 收當日預約推播 → 在行事曆看預約 → 收到詢價單 → 在 Line 或後台報價 → 客人選中 → 預約成立 → 客人到店 → 掃 QR Code 或輸入驗證碼調出訂單 → 安裝 / 保養 → 填寫履歷(可拍照、可加額外施工) → 現場收工資 → 送出記錄

### 3.3 PCM 後台視角
審核店家申請 → 審核車籍資料 → 處理退貨/瑕疵爭議 → 調整儲值金(手動或自動觸發)→ 觀察店家評價 → 處理客人否認的履歷記錄 → 處理 90 天滯留商品

## 4. 核心流程拆解

### 4.1 訂單下單流程(三種收件方式)

客人結帳時三選一:

| 收件方式 | 運費 | 後續 |
|---|---|---|
| 宅配到府 | 需付 | 寄客人家、客人自行安排安裝 |
| 超商取貨 | 需付 | 寄超商、客人取件、自行安排安裝 |
| 寄合作店家(不指定) | 需付 | 系統之後提示客人詢價選店、若成功安裝運費退儲值金 |

**關鍵設計**:下單時**不需要**選哪家店、**不需要**預約時段。下單只做一件事:決定買什麼、決定收件方式、付款。

### 4.2 詢價流程

**客人端:**
1. 貨到後在會員中心點「詢價安裝」
2. 勾選訂單中要安裝的商品(可選部分、非全部)
3. 最多可勾選 3 家合作店家
4. 填備註(例:「希望兩週內安裝」)
5. 送出詢價單、設定 48 小時截止

**系統發 Line 給每家店家(圖文卡片):**

```
🔧 新詢價單 — 王小明 MT-09
[商品圖片]
- Brembo Stylema 卡鉗組 × 1
- Öhlins 前叉套件 × 1
👉 查看商品頁
備註:希望兩週內安裝
⏰ 請於 48 小時內回報
👉 點此回報 [連結]
```

**店家端:**
- 點連結進入簡單網頁(不需登入)
- 填寫:每項工資、可接時段(最多 3 個)、備註
- 送出

**客人端看到比較表:**

| | 新莊合作店 | 桃園合作店 | 台中合作店 |
|---|---|---|---|
| Brembo | NT$1,500 | NT$1,200 | NT$1,800 |
| Öhlins | NT$2,000 | NT$2,200 | NT$1,800 |
| 小計 | NT$3,500 | NT$3,400 | NT$3,600 |
| 時段 | 週六下午 | 下週三 | 週日早上 |
| 備註 | 可現場檢查 | 需提前調貨 | 包含試車 |

**店家逾時未回**:
- 系統自動從比較表移除該店家
- 提示客人「此店家未回應、可另選其他店家或重發詢價」

### 4.3 預約流程

客人選定店家後進入預約:
1. 系統從店家報價的可接時段中讓客人選一個
2. 或客人輸入其他時段讓店家確認
3. 系統發 Line 給店家:「客人已選中您、預約 XX 時間、請確認」
4. 店家點確認 → 預約成立
5. 系統發 Line 給客人:「預約確認、記得準時到店」

**改期規則:**
- 距預約時間 ≥ 36 小時 → 客人可自行發起改期、走重新詢價的縮簡流程
- 距預約時間 < 36 小時 → 需直接聯絡店家 Line 或電話
- 店家主動改期 → 系統發 Line 通知客人

**衝堂處理:**
- 系統**不擋**店家同時段多筆預約
- 店家自己判斷接不接、可在後台回覆「此時段已排滿、請改期」
- 未來若店家需要、可在後台設定「每時段最多接 X 筆」功能(v2)

**訂單拆分多筆預約:**
- 一筆訂單可拆成多筆預約(例:5 個零件拆去 2 家店)
- 每筆預約獨立管理狀態、改期、履歷
- 商品可標記「不需預約」(客人自己裝)

### 4.4 安裝流程

客人到店時:
1. 客人出示 QR Code(訂單確認頁或 Email 都有)→ 店家用手機掃
2. 或客人唸 6 位數驗證碼 → 店家在後台首頁大輸入框輸入
3. 或店家輸入客人手機後 4 碼 → 系統列出符合的訂單、店家選
4. 系統跳出該訂單詳情(商品圖片、料號、車款、訂購日期)
5. 店家確認對應後進入履歷寫入介面

### 4.5 履歷寫入流程

店家填寫:
- 勾選訂單中實際安裝的商品(可能沒裝全部、客人當天只裝一部分)
- 新增「額外施工項目」(例:順便換機油 NT$800、調鏈條 NT$300、檢查煞車油免費)
- 上傳現場照片(選填、但有拍會顯示「有圖佐證」📷)
- 填入當前里程數
- 填入備註
- 送出

系統發 Line 給客人:

```
新莊合作店剛幫您的 MT-09 記錄了一筆安裝
- Brembo 卡鉗組
- Öhlins 前叉套件
- 額外:換機油
安裝日期:2026-05-15
里程:10,100 km
請確認是否正確?
✅ 確認   ❌ 不是我
```

客人回應處理:
- 48 小時內點「確認」→ 履歷正式寫入
- 48 小時內點「不是我」→ 記錄擱置、PCM 後台人工處理
- 48 小時未回應 → 自動轉為「已確認」(客人假如不異議就算同意)
- PCM 後台看到久置未確認記錄會主動 Line 或電話關心客人

### 4.6 儲值金回饋機制

觸發條件(全部符合才退):
- 客人下單時選「寄合作店家(不指定)」、運費已收
- 成功透過合作店家完成安裝(履歷確認寫入)
- 本次訂單至少一個商品走完整的安裝流程

退款邏輯:
- 退本次訂單運費、不另加碼
- 自動轉為儲值金、不退現金
- 系統發 Line 給客人:「恭喜您選擇合作店家安裝、運費 NT$XXX 已退為儲值金、下次消費可抵用」

⚠️ **與舊 repo backlog #8 的交互警告**(Sean 2026-04-22 標示、新 repo 需重新映射):
- 舊 backlog #8 定義「累積儲值滿 NT$100,000 自動升 premium_store」
- 本節儲值金回饋會計入累積、可能觸發升級判斷
- Phase 2 P2-10 實作前必須釐清:儲值金回饋算不算「累積儲值」?
- 若算、單純裝零件也可能意外升成店家等級、需加排除規則
- 若不算、在資料表層需區分「消費儲值」vs「回饋儲值」兩種類型

### 4.7 滯留商品處理(三階段提醒)

客人選「寄合作店家」但長期未預約 / 未取件:

第 30 天系統自動發 Line + Email:

```
您的訂單 #XXXX 已送達 XX 合作店 30 天
請盡快與店家預約安裝或改寄到府
👉 我要預約 / 改寄到府
```

同時通知店家與 PCM 後台

第 60 天系統發第二次:

```
訂單 #XXXX 已滯留 60 天
再過 30 天若無回應將視為放棄
商品會由店家寄回 PCM
```

第 90 天轉為「放棄」狀態:
- 系統通知店家:「客人放棄、請寄回 PCM、運費由 PCM 負擔、👉 列印退貨單」
- 商品回到 PCM 倉庫
- 退款先保留在 PCM、Sean 後台人工決定退現金 / 轉儲值金 / 換購他物

### 4.8 爭議處理

瑕疵 / 發錯貨 / 安裝失敗 / 收費爭議:
- 客人拍照 + 描述問題 → 發 Line 或 Email 給 PCM 客服
- PCM 後台人工判斷責任歸屬
- PCM 的錯(發錯貨 / 瑕疵 / 運送損壞)→ 退貨、運費 PCM 付、換貨或退款
- 客人的錯(買錯型號 / 車型不合 / 自己不想要)→ 不退、說明原因
- 有爭議 → Sean 親自判斷

前台**不做**自助退貨按鈕、所有退貨走客服人工管道以保品質。

## 5. 資料模型

### 5.1 vehicles(車輛主檔)

```
vehicles
├── id                 uuid PK
├── owner_member_id    uuid FK → members.id(當前車主、null 表示後台保留區)
├── make               text  (廠牌、例:Yamaha)
├── model              text  (車型口語名、例:MT-09)
├── model_official     text  (車型正式名、例:MTN890、人工確認後填)
├── year               int   (年份)
├── engine_no          text  (引擎號碼、行照 OCR 來源)
├── vin                text  (車身號碼、行照 OCR 來源)
├── plate              text  (車牌、選填)
├── color              text  (選填)
├── id_doc_url         text  (行照圖片 URL)
├── ocr_result         jsonb (OCR 原始辨識結果)
├── verified           bool  (車籍已驗證 = true)
├── verified_at        timestamp
├── verified_by        uuid  (PCM 管理員 ID)
├── deleted_at         timestamp (30 天緩衝用、null 表示未刪)
├── created_at         timestamp
└── updated_at         timestamp
```

### 5.2 vehicle_service_records(服務記錄)

```
vehicle_service_records
├── id                    uuid PK
├── vehicle_id            uuid FK → vehicles.id
├── shop_member_id        uuid FK → shops.id (寫入的店家)
├── shop_staff_id         uuid FK → shop_staff.id (操作的員工、可為 null)
├── order_id              uuid FK → orders.id (若對應訂單、null 表示店家獨立記錄)
├── is_pcm_verified       bool (有 order_id 且時間合理 = true)
├── service_type          enum ('installation' | 'maintenance' | 'inspection' | 'repair')
├── items_installed       jsonb (訂單內勾選的商品陣列)
├── extra_services        jsonb (額外施工項目 + 金額)
├── labor_cost            int  (總工資、NT$)
├── mileage               int  (當時里程 km)
├── photos                jsonb (現場照片 URL 陣列)
├── has_photo             bool (有照片 = true、顯示 📷 標記)
├── note                  text
├── customer_confirmed    enum ('pending' | 'confirmed' | 'denied' | 'auto_confirmed')
├── confirmed_at          timestamp
├── performed_at          timestamp
└── created_at            timestamp
```

### 5.3 service_pricing_requests(詢價單)

```
service_pricing_requests
├── id                    uuid PK
├── member_id             uuid FK
├── order_id              uuid FK
├── item_ids              jsonb (客人勾選要安裝的商品 ID 陣列)
├── invited_shop_ids      jsonb (客人選擇問的店家、最多 3 家)
├── note                  text  (客人備註)
├── status                enum ('pending' | 'completed' | 'expired' | 'cancelled')
├── expires_at            timestamp (48 小時後)
├── selected_shop_id      uuid (客人最終選中的店家)
├── created_at            timestamp
└── updated_at            timestamp

service_pricing_responses(店家報價)
├── id                    uuid PK
├── request_id            uuid FK → service_pricing_requests.id
├── shop_id               uuid FK
├── item_quotes           jsonb (每項工資)
├── labor_total           int
├── available_slots       jsonb (可接時段、最多 3 個)
├── note                  text
├── responded_at          timestamp
└── expired               bool
```

### 5.4 shops + shop_staff(店家帳號結構)

```
shops(店家主帳號)

```

### 5.5 maintenance_reminders(保養提醒)

```
maintenance_settings(客人的保養設定)
├── id                    uuid PK
├── vehicle_id            uuid FK
├── item_key              text ('engine_oil' | 'brake_pad_front' | 'chain' | ...)
├── enabled               bool
├── interval_km           int  (客人自訂的週期、覆蓋系統預設)
├── last_service_mileage  int  (上次服務時里程)
├── last_service_date     date
└── updated_at            timestamp

mileage_logs(里程累積資料點)
├── id                    uuid PK
├── vehicle_id            uuid FK
├── mileage               int
├── source                enum ('customer' | 'shop_install' | 'shop_maintain')
├── source_ref_id         uuid (若來自店家記錄、指 vehicle_service_records.id)
├── recorded_at           timestamp
└── created_at            timestamp
```

系統內建基範保養項目:
- engine_oil(機油):3,000 km
- brake_disc_front(前煞車碟盤):20,000 km
- brake_disc_rear(後煞車碟盤):20,000 km
- brake_pad_front(前煞車來令片):10,000 km
- brake_pad_rear(後煞車來令片):10,000 km
- chain(鏈條保養):1,000 km
- air_filter(空氣濾芯):10,000 km
- spark_plug(火花塞):10,000 km

### 5.6 member_discount_overrides(個人化折扣)

```
member_discount_overrides
├── id              uuid PK
├── member_id       uuid FK
├── brand_id        uuid FK (null 表示整體加成、所有廠牌都適用)
├── discount_rate   decimal (例:0.05 = 5%)
├── mode            enum ('override' | 'extra')
├── note            text (例:「長期合作 VIP」)
├── created_by      uuid (執行的 admin)
├── created_at      timestamp
└── updated_at      timestamp
```

## 6. 防偽機制(三道關卡)

### 6.1 第一道:車籍驗證(防客人亂填)

客人登記車輛流程:
1. 填車輛資料(車型、年份、車身號碼、引擎號碼)
2. 上傳行照照片
3. 系統 OCR 自動辨識(預計用 Google Vision API、每張約 NT$0.05)
4. 辨識欄位:車身號碼、引擎號碼、出廠年份
5. 系統比對 OCR 結果 vs 客人填的資料
6. 送 PCM 後台人工審核(同時處理「車型口語名 vs 正式名」的對應)
7. 審核通過 → 車輛標「🔒 已驗證車籍」、發 Line 通知客人

未驗證的車輛、履歷可信度標為「低」。

**關於車型對應**:客人俗稱「MT-09」、行照可能是「MTN890」或「RN69J」。這個對應表不存在現成資料庫、PCM 後台需累積成內部資產。累積後可供 Phase 3+ 開放給其他業者(商業延伸點)。

### 6.2 第二道:訂單綁定 + 時間合理性(防店家造假)

PCM 認證標記條件(全部符合):
- 這筆訂單存在於 PCM
- 訂單是這個客人的
- 安裝日期在訂單出貨後 30 天內(時間合理)

任一不符 → 只能標「📋 店家記錄」、可信度較低。

### 6.3 第三道:客人確認(防串通造假)

機制:
- 店家寫入後、系統發 Line 給客人
- 客人在 48 小時內按「✅ 確認」或「❌ 不是我」
- 48 小時未回 → 自動轉為「已確認」
- 客人按「不是我」→ 記錄擱置、PCM 後台人工處理
- PCM 後台可主動 Line 或電話關心久置未確認記錄

### 6.4 履歷可信度分級

| 標記組合 | 條件 | 可信度 |
|---|---|---|
| ✅ PCM 認證 + 🔒 車籍已驗 + 📷 有圖佐證 | 所有驗證 | 最高 |
| ✅ PCM 認證 + 🔒 車籍已驗 | 驗證 + 訂單綁定 + 客人確認 | 高 |
| 📋 店家記錄 + 🔒 車籍已驗 + 📷 | 店家獨立記錄 + 行照驗證 + 照片 | 中 |
| 📋 店家記錄 + 🔒 車籍已驗 | 店家記錄 + 行照驗證 | 中低 |
| 📋 店家記錄(未驗車籍) | 僅店家記錄 | 低 |

## 7. 二手車履歷邏輯

### 7.1 交接簡化版(核心原則:A 不用動作)

A 賣車給 B:
- 系統在 A 帳號下把這台車標記「已讓與」、移到「我曾經擁有的車輛」分頁
- A 永遠看得到舊履歷(唯讀)、但不能編輯

B 加新車:
- 輸入車身號碼
- 系統偵測「這台車有前任車主的履歷」
- 顯示「這台車在系統內有 X 筆前任車主的記錄、您的履歷將從空白開始」
- B 同意接手 → B 從空白開始累積自己的記錄

### 7.2 店家可查完整歷史

店家輸入車身號碼 → 看到完整歷史(含前車主時代 + 新車主時代、跨所有車主)。

這是 PCM 生態系的核心價值:履歷不公開、僅合作店家可查、這讓店家有「獨家資訊」誘因加入 PCM。

### 7.3 前車主個資隱藏

新車主 B 看不到 A 的姓名、手機、地址等個資。只顯示「前車主」+ 安裝記錄的技術內容。

### 7.4 車輛刪除緩衝機制

客人刪車不是真的刪、是標「deleted_at」:
- 30 天內加回同車身號碼 → 系統提示「您曾登記過這台車、是否恢復履歷?」、可一鍵接回
- 30 天以上未加回 → 資料轉後台保留區、需聯絡 PCM 客服恢復

## 8. 店家端功能

### 8.1 店家申請與審核(兩軌並行)

軌道 A:店家主動申請
- 在官網填申請表(店名、地址、統編、營業執照照片、聯絡人、Line OA ID)
- PCM 後台收到通知
- 人工審核通過後開帳號、發 email 給店家

軌道 B:PCM 主動建立
- Sean 談下合作的店家、直接在後台開帳號
- 省掉申請流程

### 8.2 帳號結構:主帳號 + 可選子帳號

一家店一個主帳號(owner):
- 主帳號可管理子帳號(新增 / 刪除 / 權限)
- 主帳號可做所有操作

子帳號(staff):
- 可選擇性開給員工
- 預設可寫入履歷記錄
- 預設不能管理其他子帳號(只有 owner 可以)
- 每筆操作記錄「哪個員工做的」(shop_staff_id)

### 8.3 登入保護

- 密碼登入即可(最簡單、店家上手最容易)
- 異常登入偵測:
  - 短時間多次登入失敗 → 通知 Sean + 店家主帳號
  - 從不熟悉地點登入 → 通知 Sean + 店家主帳號
- 這個是被動保護、不增加店家日常負擔

### 8.4 行事曆(店家核心工具)

顯示三種來源的事件:
- 🟦 pcm_booking:PCM 預約(不可刪、可改狀態)
- 🟩 shop_self:店家自加預約(可編輯、可刪)
- 🟨 shop_private:店家私人事項(可編輯、可刪)

檢視模式:
- 月檢視(看全月)
- 週檢視(看本週)
- 單日檢視(看當天詳情)

互動:
- 點日期進去看該日所有事件
- 可直接在事件上改狀態(已完成 / 客人未到 / 需重新安排)
- 可手動新增事件(填時間、標題、備註)

### 8.5 每日 Line 推播

每天早上 9 點自動發 Line 圖文卡片給店家:

```
🔧 [店名] 今日預約

10:00 - 王小明 MT-09
- Brembo 卡鉗組
- 訂單 #1234

14:00 - 陳大華 Panigale V4
- Öhlins 前叉套件
- 訂單 #1235

👉 查看完整行事曆 [連結]
```

### 8.6 店家公開頁面(作品集)

每家合作店家有自己的公開頁面、客人可瀏覽:

顯示內容:
- 店名、地址、電話、營業時間
- 累計數據:服務車主數、完成筆數、合作月數、擅長車型
- 最近完成的作品(時間排序)
  - 每筆顯示:日期、車款、安裝商品、有無照片標記
  - PCM 認證的記錄 → 顯示商品頁連結(點進回 PCM 商品頁)
  - 店家獨立記錄 → 無商品連結(只顯示項目名稱)

**不顯示**:工資金額、客人姓名、星級評分。

### 8.7 商品頁面反向連結(流量互導)

PCM 商品頁新增「這個商品誰在裝」區塊:
- 顯示最近有多少人裝這個商品
- 列出有成功安裝案例的合作店家(點進店家頁)
- 從商品頁可直接進入詢價流程

這是雙向流量導流:店家頁面帶流量給商品、商品頁帶流量給店家。

### 8.8 UX 設計原則

店家端的設計假設:店家老闆「不太會用電腦手機」。

具體要求:
- 字要大(最小 16px、標題 20px 以上)
- 按鈕要大(最小 44x44 px、手指好點)
- 一個畫面只做一件事
- 確認步驟要清楚(送出前顯示摘要、按確認才送出)
- 手機直式優先、其次平板、其次桌機

響應式設計斷點:
- 375px 手機:單欄、卡片式
- 768px 平板:雙欄可並排
- 1920px 桌機:完整行事曆 + 側欄

### 8.9 收費模式

初期:完全免費(PCM 不向店家收任何費用)。

未來規模大了再考慮:
- 月費(合作費)
- 成交抽成(與「PCM 不碰工資」的原則需重新權衡)
- 升級方案(基本免費、進階收費)

現階段**不寫進系統**、只在 PRD 記錄作為未來討論起點。

## 10. 通訊機制

### 10.1 只用 Line + Email、不用簡訊

成本考量:
- Line 官方帳號免費方案 200 則/月
- 超過後每則約 NT$0.2
- 簡訊每則 NT$1-2、成本過高

⚠️ **v0.1 review 項目**:Phase 2 啟動前需重算 Line OA 月成本上限、依店家數與預約頻率規模估總額

### 10.2 Line Bot 詢價報價流程

用「發連結 + 點網頁」的做法(做法 B)、不做「Bot 內直接互動」(做法 A):
- 店家收 Line 訊息 → 點連結 → 開簡單網頁 → 填表送出
- 不用 Bot 處理複雜互動邏輯、開發成本低、店家體驗穩定

### 10.3 客人加官方 Line

客人下單時如未加 PCM 官方 Line:
- 顯示 QR Code「加入 Line 以接收通知」
- 未加 Line 時自動 fallback 到 Email 通知

## 12. 商業模式考量

### 12.1 金流分工
- PCM:只賺零件差價
- 店家:只賺工資 + 額外施工
- 雙方利益不衝突、不爭奪同一筆錢

### 12.2 儲值金回饋的誘因設計
- 客人選「寄合作店家」+ 完成安裝 → 運費退儲值金
- 鼓勵客人走 PCM 生態系、不用強迫
- 儲值金只能在 PCM 用、鎖住客人回流

### 12.3 飛輪效應
- 客人多 → 店家看到機會 → 店家加入多
- 店家多 → 商品頁有真實案例 → 客人看到後更願意買零件
- 零件賣得多 → PCM 商品豐富 → 更多客人加入
- 履歷只開放店家查 → 店家有獨家資訊價值 → 更願意深度合作
