## §2. Round 1:engineering:code-review

**視角:** security / correctness / 維護性
**聚焦點:**
- matchFitment 邏輯是否正確覆蓋 4 種狀態(無年份 / 範圍 / 開放式 / 單年)
- yearEnd null 處理(Infinity 推算)是否會在 edge case 撞坑
- contract.ts 純架子是否有 type signature 問題
- test case 是否漏掉重要 boundary
- private method 透過 listByFitment 間接測試的覆蓋率

### 2.1 Critical

#### F1:[Critical] `packages/ports/src/index.ts:14` runtime re-export 帶入 vitest 依賴有洩漏 production 風險

```ts
export { runProductRepositoryContract } from './IProductRepository.contract';
```

contract.ts 第 1 行 `import { describe, it } from 'vitest';`。`@pcm/ports` 被 storefront / sync-engine 等 production app import 時、若 tree-shaking 不生效(pnpm workspace + esbuild + 沒標 `sideEffects: false`、或 vitest 模組自身有 side effects),vitest 會被拉進 production bundle。

**建議:** 改 subpath export `@pcm/ports/contract`(package.json `exports` 加 `"./contract": "./src/IProductRepository.contract.ts"`),或拆獨立 package `@pcm/ports-contract`。註解寫「tree-shaking 友好」是樂觀假設、不能當保證。

### 2.2 Major

#### F2:[Major] `IProductRepository.contract.ts:71` `describe('matchFitment year-range', ...)` 暴露私有實作名

`matchFitment` 是 InMemoryProductRepository 的 private method、不在 IProductRepository 介面字面。Contract 級不該知道 adapter 內部方法名、否則 main-b SupabaseProductAdapter 用完全不同實作(PG range query、無 matchFitment)時、describe 名變誤導。

**建議:** 改 `describe('listByFitment year-range matching', ...)` 或 `describe('listByFitment fitment-matching', ...)`。it.todo 字面對應改。

#### F3:[Major] `InMemoryProductRepository.test.ts:148` 同樣命名洩漏

`describe('matchFitment year-range', ...)` 在 in-memory test 出現可接受(這層確實有 matchFitment private method)、但既然 contract 級也用同名、語意混淆。**建議改與 contract 對齊** `listByFitment year-range matching`、保持一致語意層級。

### 2.3 Minor

#### F4:[Minor] `InMemoryProductRepository.ts:111-112` yearEnd 三元 + `??` 混用可讀性低

```ts
const actualEnd = actual.yearEnd === null ? Infinity : actual.yearEnd ?? actual.yearStart;
```

三層語意(null / undefined / number)擠一行。雖邏輯正確(`=== null` 先檢查、避免 `??` 吞 null)、reader 需理解 `??` 不會吞 null、加上 `??` 的優先級規則。**建議:** 抽 helper `resolveEnd(start, end)` 或拆 if/else。

#### F5:[Minor] `InMemoryProductRepository.ts:88-100` JSDoc 規則 3 漏寫 spec 端對稱處理

註解只寫 actual.yearEnd 三狀態、沒明寫 specEnd 也走同樣處理(程式碼第 112 行 specEnd 邏輯)。**建議補:** 「actual / spec 兩端對稱處理 yearEnd null/undefined」。

#### F6:[Minor] 漏測 boundary cases(4 個 yearRange test 不夠)

- 單年 actual(yearEnd=undefined)+ 範圍 spec → 應 match
- 單年 actual + 單年 spec、相同年 → 應 match
- 單年 actual + 單年 spec、相鄰年 → 不應 match
- 邊界 inclusive(actual.yearEnd=2020 vs spec.yearStart=2020)→ 應 match(驗 `≤` 含等於)
- 空 fitments[] → some() 回 false、不 match
- 多 fitments[] OR(product 有 fit1+fit2、spec 只 match fit2)→ 應 match
- yearEnd null 開放式 + spec 早於 yearStart(actual=2025+ vs spec=2024)→ 不應 match
- actual 無年份的鏡像(test case 3 只測 spec 無年份、沒測 actual 無年份)

#### F7:[Minor] `InMemoryProductRepository.ts:103-104` motoBrand / modelCode strict equality case-sensitive

`'Yamaha' !== 'yamaha'` 會 silent miss。對齊 ADR-0004 wrs Q1=A1 字面(自由字串)、但實作是 strict equality、case 不一致會 false negative。**建議:** 對齊 searchByKeyword 的 `.toLowerCase()` pattern、或 sync engine 上架時 normalize。屬資料層議題、現階段標 Minor。

#### F8:[Minor] `InMemoryProductRepository.ts:99` 範圍重疊判定無 yearStart > yearEnd 防呆

若資料 yearStart=2024, yearEnd=2018(資料異常)、邏輯不報錯、回 false negative。屬資料層責任、但無 log / no warning。

#### F9:[Minor] `listByFitment` 結果無排序契約

`Array.from(this.products.values()).filter(...)` 回 Map insertion order、不穩定(雖實際 V8 stable insertion、契約上無保證)。caller 若期待按某順序(例 yearStart desc)、需自行 sort。

### 2.4 Nit

#### F10:[Nit] `IProductRepository.contract.ts:44` `void factory;` trick 雖正確但 quirky

替代寫法:`function runProductRepositoryContract(_factory: () => IProductRepository)` 或 eslint-disable-next-line + 註解。**建議:** 維持現況、註解已清楚說明、不必改。

#### F11:[Nit] `InMemoryProductRepository.ts:101` `actual` / `spec` 變數名不直觀

讀者首次看 `matchFitment(actual, spec)` 易把 actual 當「實際結果」、其實是 product 端的 fit。**建議:** `productFit` / `queryFit` 或 `candidate` / `query`。

### 2.5 Round 1 What Looks Good

- contract.ts JSDoc 對齊 ADR-0003 §3.3 / testing-strategy §3.4 / backlog #86 三軸合一、reference 完整
- `actual.yearEnd === null ? Infinity : actual.yearEnd ?? actual.yearStart` 順序正確(先 `=== null` 檢查、避免 `??` 吞 null 撞 silent bug)
- type narrowing 後不需 `!` assertion 的註解(line 107)降低未來誤加 non-null assertion 機率
- 4 個 yearRange test 對齊 Sean 列的 4 種狀態(範圍重疊 / null 開放式 / 無年份 / false-positive)
- structuredClone 在 save 對齊 M-1-02-audit C1 defensive copy
- contract framework factory signature 預留、main-b 落地時不需重構介面(Sean Q1=A3「純架子」拍板穩定)
- searchByKeyword empty query 回空 Paginated 對齊 M-1-02-audit M2

## §3. Round 2:simplify

**視角:** reuse / quality / efficiency
**聚焦點:**
- matchFitment 邏輯簡潔度
- contract.ts framework 結構 reuse 機會
- test case 重複模式
- yearEnd null vs undefined 雙語意 type guard

### 3.1 Reuse 視角

#### F12:[Major] `InMemoryProductRepository.ts:111-112` `resolveEnd` 對稱邏輯可抽 helper

```ts
const actualEnd = actual.yearEnd === null ? Infinity : actual.yearEnd ?? actual.yearStart;
const specEnd = spec.yearEnd === null ? Infinity : spec.yearEnd ?? spec.yearStart;
```

兩行對稱、僅 actual / spec 換、屬 copy-paste with slight variation。可抽:

```ts
const resolveEnd = (start: number, end: number | null | undefined): number =>
  end === null ? Infinity : end ?? start;
```

兼解 yearEnd null vs undefined 雙語意 type guard(Sean 聚焦點之一)。SupabaseProductAdapter(main-b)也得處理 null/undefined → PG range bound、抽到 `packages/domain/src/catalog/year-range.ts` 共用更佳。

**建議時機:** main-b 落地時抽(不在 prep 階段抽、避免過早抽象、Sean Q1=A3「純架子」精神)。

#### F13:[Minor] `InMemoryProductRepository.ts` `Array.from(this.products.values()).filter(...)` 在 4 method 重複

listByCategory / listByBrand / listByFitment / searchByKeyword 各一次。可抽 `private all(): Product[]` 或 `private query(predicate)` helper。**建議:** 不抽(4 處、抽不大降複雜度、可能引入 indirection),YAGNI 級 Nit。

#### F14:[Nit] `IProductRepository.contract.ts` 6 method describe + it.todo 結構重複

每個 `describe(method, () => it.todo('main-b SupabaseProductAdapter 落地時實作:...'))` 模式相同。可抽 `itTodoMainB(method, desc)` helper、但 main-b 落地時各 it.todo 變不同 it body、helper 反而 unwrap、屬過早抽象。**建議:** 不抽。

### 3.2 Quality 視角

#### F15:[Major] `InMemoryProductRepository.ts:111` ternary + `??` 混用可讀性低(同 F4 quality 視角再命中)

`actual.yearEnd === null ? Infinity : actual.yearEnd ?? actual.yearStart` 三狀態壓一行、reader 需理解「為何 `??` 不會吞 null」(答:先 `=== null` 已濾掉)。

抽 `resolveEnd` helper 後、語意呼之欲出。與 F12 同一 fix。

#### F16:[Major] `IProductRepository.contract.ts:71` Leaky abstraction(同 Round 1 F2 重疊命中)

`describe('matchFitment year-range')` 在 contract 級暴露 InMemory 私有實作名 — 屬 simplify checklist「Leaky abstractions: 暴露內部細節」。Round 1 F2 / Round 2 同視角抓到、Critical 確認。

#### F17:[Minor] `InMemoryProductRepository.test.ts:148-216` 4 個 yearRange test setup 重複

每 test 模式:
```ts
const product = createFakeProduct({...});
const repo = new InMemoryProductRepository([product]);
const result = await repo.listByFitment({...});
expect(result).toHaveLength(N);
expect(result[0]?.id).toBe('...');
```

可考慮 `it.each([...])` table-driven、或抽 helper `expectFitmentMatch(fitments, query, expectedIds)`。

**建議:** 不急抽。table-driven 在 `motoBrand / modelCode / yearStart / yearEnd / null / undefined` 多維度時可讀性可能變差、4 個 test 各自表達意圖反而清楚。標 Nit、main-b 補 boundary cases(F6)擴到 8-12 個 test 後再考慮。

#### F18:[Minor] `InMemoryProductRepository.ts:107` 註解 vs JSDoc 重複

```ts
// 任一邊無年份 → match(narrowing 後 yearStart 為 number、不需 ! assertion)
if (actual.yearStart === undefined || spec.yearStart === undefined) return true;
```

「任一邊無年份 → match」JSDoc 規則 3 已寫、註解 inline 重複。但「narrowing 後 yearStart 為 number、不需 ! assertion」是 WHY、防未來啟 noUnusedLocals 撞、保留價值高。**建議:** 砍前半「任一邊無年份 → match」、保留後半 narrowing 註解。

### 3.3 Efficiency 視角

#### F19:[Critical] `packages/ports/src/index.ts:14` vitest 拉進 production payload(同 Round 1 F1 重疊命中)

efficiency 視角看 = bundle size + 啟動時間。Round 1 從 security / correctness 視角抓到、Round 2 從 efficiency 視角同樣命中。**Critical 確認。**

#### F20:[Minor] `InMemoryProductRepository.ts:77-81` searchByKeyword 每 product 兩次 toLowerCase()

```ts
p.name.toLowerCase().includes(q) || p.description.toLowerCase().includes(q)
```

N products × 2 toLowerCase calls / search。對 200 SKU InMemory 無虞、production 走 PG ILIKE 不走這條。**建議:** 不改、屬 InMemory 真實作可接受成本。

#### F21:[Nit] `InMemoryProductRepository.ts` `Array.from(this.products.values())` 每次 list 建新 array

對 InMemory 200 SKU 無虞。production Supabase 不走這條。**建議:** 不改。

### 3.4 Round 2 What Looks Good

- matchFitment 邏輯密度合理(13 行核心 + 註解)、無 efficiency 浪費
- structuredClone 在 save 是 InMemory 模擬 production immutability 必要代價、不過度

## §9. 教訓與規範產出

本 audit 產出 2 條規範進 lessons-learned.md §12:

### §12-1 test framework re-export 必走 subpath

純 test framework(含 vitest 等 devDependency)不可從 main entry index.ts re-export。必走 subpath export(`exports` field `./contract` 子路徑)。

來源:F1 / F19(雙視角 Critical 同向命中)。

### §12-2 contract 命名以 port public method 為錨

contract 級 describe 名必以介面公開 method 為錨、不暴露 adapter 內部 helper / private method 名。同 method 多維度測試用嵌套 describe 表達層次。延伸 testing-strategy.md §3.4「樣板不外洩」精神。

來源:F2 / F3 / F16(雙視角 Major 同向命中)。
