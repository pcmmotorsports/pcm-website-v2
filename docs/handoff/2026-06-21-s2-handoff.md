## ① 做了什麼（白話）

3DS 刷卡完成後，客人會被銀行導回「訂單結果頁」。但實測發現：銀行雖然授權成功（客人收到刷卡簡訊），TapPay 的查詢 API 有**幾秒同步延遲**，所以那一刻網站常常還查不到「成立」→ 客人只看到「處理中」，得自己手動刷新才看到訂單成立。

S2 補的就是這段體驗：
1. **背景自動輪詢**：停在「處理中」時，網站在背景每隔幾秒自動問一次「這筆成立了沒」，一旦成立就**自動跳成功頁**（客人無感、不用手動刷新）。
2. **處理中文案安撫**：把「處理中」頁的文字改成對齊設計決策包 §5.5 的三個重點 —— 已收到付款、銀行授權成功是正常的、**請勿重複付款**。

技術細節（下層）：
- 新增一支「會員查自己訂單付款狀態」的 API（`GET /api/orders/[orderId]/payment-status`）。
- 新增一個前端輪詢小元件 `PollOrderStatus`，掛在「已歸屬 + 處理中」的結果頁上。
- 改 callback 結果頁：處理中變體掛輪詢 + 文案拆成兩種（已歸屬可斷言 vs 泛用態中性，見下）。

## 🔴 安全紅線（已鎖死、codex + code-reviewer 雙核 PASS）

- **IDOR own-only**：輪詢端點雙軸防線 —— RLS `orders_select_own` + 應用層 `.eq('customer_user_id', userId)` + `getUser()` 驗 JWT。**別人偽造你的訂單號也只查得到自己的單**（查無→404）。
- **經銷價零洩漏**：端點只讀 `payment_status` 單欄、**回應只含 `{status:'paid'|'pending'}`**（零金額、零單號、零任何價格欄）。
- **fail-closed**：查不到/超時/非 paid 一律**不偽 paid 不偽 failed**；前端只在明確回 paid 才跳成功頁，否則維持安撫文案。
- **不碰結算邏輯**：S2 端點**只讀** payment_status、**不呼叫 settleCharge**；`settle-charge.ts` 零變更（git diff 空）。

**Q1 — 輪詢要「只看狀態」還是「主動去結算」？（最重要、影響功能真正有沒有效）**
- **A（已落地、default、最保守）**：輪詢端點**只讀** `orders.payment_status`，不主動結算。靠背景（webhook / sweeper 排程）把訂單推成立後，輪詢才看得到。
  - 🔴 **誠實缺口**：實測 callback 當下 TapPay 查詢還沒同步 → 那一刻的背景結算也常失敗；而現在的 sweeper 排程是**每天一次**（要到 S6 才升頻）。所以 default A 下，「幾秒內無感自動成立」**不保證**，本機/現況多半會輪詢到超時（仍 fail-closed 顯安撫文案 + email）。**A 只保證「輪詢機制 + 安全」本身正確，不保證即時成立。**
- **B（備選、才真達成設計包 §5.1「幾秒無感成立」）**：輪詢端點在確認是本人單後**主動呼叫 settleCharge**（去 TapPay 反查、同步後立刻成立）。代價：① 輪詢期間會重打 TapPay 查詢 API（需加「同一張單 N 秒內只查一次」的節流防被多分頁/重整打爆）② 端點責任變重（要再過 codex 重審）。
- **我的建議**：若你要客人「下單完幾秒內自動看到成立」的體驗，選 **B**（但要排 throttle，約多半天工）；若可接受「靠 email 通知 + 之後刷新看得到」、先求穩，留 **A**、把即時性交給 S6 把 sweeper 升頻。**端點已預留 B 的形狀，切換成本小。**

- `orders.payment_status` enum **沒有 failed 值**（失敗是 `payment_charge_attempts` 層的狀態）。所以「失敗」要顯示，**得讀 attempt 狀態**（碰 payment 讀路徑 = 🔴 鐵則 12，需 codex 雙關卡）。
