## 5. 🔴 B 硬前置 ＝ durable per-order throttle（不可省、安全紅線、N-S2-b）

- **問題**：pending 單每次輪詢都呼 settleCharge → settleCharge step3 `recordQuery` **每次打 TapPay Record**；13 次/頁載入 × 多分頁/重整 ＝ 會員可 spam 打爆 Record 查詢額度。settleCharge step2 只對「已 paid」短路（不打 Record），**pending 不短路**。
- **要求**：同一 `orderId` 在 throttle 窗（建議數秒、plan 定）內已 settle 過 → **skip settleCharge/Record、直接讀 `payment_status` 回**（等同 A 版行為、零放大）。
- **durable**：serverless 多實例 → **in-memory 不可靠、不可用**，必 durable（DB）。
- **候選＝重用 4a-2 `payment_charge_attempts.next_settle_at` / `settle_attempt_count`**（sweeper 已用、durable）。🔴 **但 plan + codex K1 必先驗證不與 sweeper 對 `next_settle_at` 的「退避排程」語意衝突**（throttle＝「最近結算過、N 秒內別再打」;sweeper 退避＝「下次重試時點」，兩者語意不同、共用同欄可能互踩）。若衝突 → 改**獨立 throttle 欄/機制**（＝動 migration＝鐵則 8 另需 plan + Sean 批 + db push 評估，先 raise Sean）。
- **冪等 + 競態**：throttle 讀寫須對並發輪詢安全（多分頁同時打）；settleCharge 本身冪等（Record 權威 + markCharged/confirm 冪等），throttle 是「省 Record 呼叫」非「保正確」，但不得因 throttle 誤鎖住正常結算。

## 6. 紅線（S2 已有、S2b 必保留 ＋ 新增）

- **IDOR own-only**：settleCharge 呼叫必在 own-only 歸屬閘「**後**」（getUser 驗 JWT + `.eq('id',orderId).eq('customer_user_id',userId)`）；歸屬失敗（401/404）**硬 early-return、絕不呼 settleCharge**（偽造他人 orderId 不得觸發任何結算）。
- **經銷價/PII 零洩漏**：回應仍只 `{status:'paid'|'pending'}`；settleCharge 回的 displayId/金額**不入回應**；400/401/404/500 維持 null body + no-store。
- **fail-closed**：settleCharge throw / 回 pending → 端點仍回 `{status:'pending'}`（不偽 paid 不偽 failed）；端點包 try/catch、不 500 不偽。
- **不改 settleCharge 內部**：`packages/use-cases/src/settle-charge.ts` **git diff 仍須空**；S2b 只「呼叫」它、不碰 classifyRecordStatus / recordMatchesOrder / 弱識別窗 / 任何結算邏輯。
- **🔴 新：throttle 防 Record 放大**（§5）。
